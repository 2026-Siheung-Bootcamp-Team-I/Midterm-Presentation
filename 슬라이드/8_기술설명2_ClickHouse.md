# 4. 기술 설명 2-2 — ClickHouse (대용량 데이터 저장)

## 역할
- 모든 원본 로그를 저장해두고 빨리 찾아주는 **대용량 창고**
- 나중에 조사하거나 통계를 낼 때 쓰기 위해 빠짐없이 모아둠

> **Kafka Streams와의 차이**: 실시간 위험 판단 = Kafka Streams / 모든 기록 저장·조회 = ClickHouse (서로 독립적)

## 왜 빠른가?
- 데이터를 행이 아닌 **열(칼럼) 단위로 저장**
- "○○ 항목에 ○○이 몇 개 있어?" 같은 질문에 필요한 항목만 읽어 계산 → **통계·집계가 매우 빠름**
- 같은 종류의 데이터끼리 모여 있어 압축 효율도 높음
- Kafka Topic을 직접 구독해 자동으로 데이터를 가져오는 **Kafka 연동 기능 내장**

## 사용 이유
- OSquery/Zeek는 짧은 시간에도 대량의 로그를 쏟아냄 → 일반 저장소보다 **대량 데이터를 빨리 받는 저장소** 필요
- Kafka Streams는 판단 후 지나간 데이터를 버림 → **필요할 때 찾아볼 기록 창고** 필요
- React 대시보드의 이벤트 추이 그래프를 일반 DB보다 **훨씬 빠르게 계산·제공**

## 운영 시 핵심 포인트
- **ORDER BY**: 자주 검색하는 조건(호스트, 시간)으로 정렬 키 지정
- **PARTITION BY**: 날짜별로 나눠 저장 → 오래된 데이터 통째로 삭제 용이
- **TTL**: "90일 지나면 자동 삭제" — 로그 무한 적재 방지에 필수
- INSERT는 한 건씩이 아니라 **1000건씩 몰아서** 넣기
- 수정/삭제·JOIN·단건 조회에는 약함 → **로그는 쌓기 전용**으로 사용
