# 4. 시스템 아키텍처 — 데이터 파이프라인 & 전체 구조

## 전체 데이터 흐름

```
OSquery / Zeek (로그 수집)
        │
        ▼
      Kafka (events 토픽)
        │
        ├──▶ Kafka Streams ── 실시간 여과(위험 판별) ──▶ Kafka (alerts 토픽) ──▶ Spring Boot ──▶ Slack / 이메일 알림
        │            │
        │            └──▶ 여과된 데이터 ClickHouse 저장
        │
        └──▶ ClickHouse (모든 원본 로그 저장)
                     │
                     ▼
              Spring Boot (API 서버) ──▶ React 대시보드
```

## 단계별 설명

1. **로그 수집** — OSquery, Zeek를 통해 실시간 로그를 입력받은 후 출력값을 Kafka로 입력
2. **이벤트 분기**
   - 2-1. 이벤트 발생 시 해당 로그를 **Kafka Streams**로 입력 → 데이터 여과 과정(실제 위험 상황인지 판별)
   - 2-2. 이벤트 발생 시 해당 로그를 **ClickHouse**에 입력하여 저장
3. **처리 결과 전달**
   - 3-1. Kafka Streams에서 여과된 데이터를 ClickHouse에 저장
   - 3-2. Kafka Streams에서 다시 Kafka로 데이터를 전송하여 **알림 기능** 구현
4. **알림** — Spring Boot의 경고 알림 기능을 활용해 **Slack 메시지·이메일** 형태로 전송
5. **시각화** — ClickHouse의 데이터를 Spring Boot에서 React가 요청하는 규격(API)으로 반환 → React에서 화면 출력
